### About the Book

Sarah Richards led the content design practice at GOV.UK and wrote the book that defined the discipline. *Content Design* is the definitive text on treating content as a design practice — evidence-based, user-centered, and measurable.

### Core Thesis

Content design is not "writing nicely." It's designing content based on evidence of what users need, in a format they can understand, at the time they need it. Sometimes the best content design is not writing at all — it's a video, a calculator, a form, or a map.

### Key Principles

**1. Start with evidence, not opinions.**
- Use analytics to find what users search for
- Use call center logs to find what users struggle with
- Use usability testing to find what users misunderstand
- Never write content because "we've always had this page"

**2. Content is not always text.**
- Sometimes the best content is a calculator (tax estimates)
- Sometimes it's a map (find a location)
- Sometimes it's a step-by-step form (apply for something)
- The content designer's job is to choose the right format, not just write well

**3. Pair writing produces better content.**
- Subject matter expert provides accuracy
- Content designer provides clarity and structure
- Together they produce content that is both correct and understandable
- This resolves the "but it needs to be technically precise" objection

**4. User needs drive everything.**
The user need format:
- "As a [type of user], I need to [do something], so that [outcome]"
- Every piece of content must map to at least one user need
- If no user need exists, the content shouldn't exist

### The Content Design Process

1. **Discovery:** Research user needs (analytics, search data, user interviews, call center analysis)
2. **Alpha:** Create prototypes with real content (not Lorem ipsum)
3. **Beta:** Test with real users, iterate on content and design together
4. **Live:** Publish, measure, continue iterating
5. **Retirement:** Remove content that's no longer needed (content has a lifecycle)

### Richards' Key Frameworks

**The "reading age" argument:**
- Target a reading age of 9 for universal comprehension
- This isn't dumbing down — research shows that even experts prefer simpler language
- High reading levels don't signal intelligence — they signal poor editing

**The "content lifecycle":**
- Content is born (published), lives (used), and dies (retired)
- Content that isn't maintained becomes harmful — outdated info is worse than no info
- Schedule regular content audits

**The "format decision tree":**
Before writing, ask:
1. Does this need to exist? (evidence of user need)
2. What format best serves the need? (text, form, tool, video)
3. If text, what structure? (guide, list, FAQ, step-by-step)
4. How will users find this? (search, navigation, contextual link)

### Actionable Takeaways

- **Audit before you write.** Most organizations need less content, not more.
- **Measure content success.** Task completion rate > page views.
- **Content needs a retirement plan.** Set review dates when you publish.
- **Pair writing is non-negotiable** for technical or specialized content.
- **"We need a page about X" is not a user need.** Challenge every content request with "What user need does this serve?"
- **Format is a content design decision.** Sometimes the answer isn't better words — it's a different format entirely.
